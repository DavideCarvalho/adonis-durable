The `queue` driver carries steps over [`@adonisjs/queue`](https://github.com/adonisjs/queue) â€” the path for steps that run in another process. It rides any `@adonisjs/queue` adapter (Redis, a database adapter, or the fake adapter in tests), so it reuses the queue infrastructure you already run.

It ships **inside `@adonis-agora/durable`** â€” there's nothing extra to install. `@adonisjs/queue` is an optional peer, imported lazily only when you select the `queue` transport, so install and configure it where you use the driver:

```bash
npm i @adonisjs/queue
node ace configure @adonisjs/queue
```

You define the adapter and its host/credentials **once**, in `config/queue.ts` (the standard `@adonisjs/queue` place), as a named connection. The durable `queue` transport then just references that connection **by name** â€” exactly like `stores.lucid({ connection })` or `stores.redis({ connection })`:

```ts title="config/queue.ts"
import { defineConfig, drivers } from '@adonisjs/queue'

export default defineConfig({
  default: 'redis',
  adapters: {
    redis: drivers.redis({ connectionName: 'main' }), // @adonisjs/redis connection
  },
})
```

## How it works

- `ctx.call` dispatches a task to a **per-group** queue: `<prefix>:tasks:<group>`.
- A worker for that group consumes it, runs the handler, and adds the result to the shared `<prefix>:results` queue.
- The engine-side instance consumes results and checkpoints them.
- Liveness flows on `<prefix>:heartbeats`; cross-instance control on `<prefix>:control`.

The default `prefix` is `durable` and the default `pollIntervalMs` is `200`.

Run **one instance engine-side** (consumes results) and **one per worker process** (registers handlers for its group):

```ts title="config/durable.ts (engine side)"
import { defineConfig, transports } from '@adonis-agora/durable'

export default defineConfig({
  transport: 'queue',
  transports: {
    // References the connection named in config/queue.ts. Omit `connection` to
    // use that file's `default`.
    queue: transports.queue({ connection: 'redis' }),
  },
})
```

Run **one instance engine-side** (consumes results) and **one per worker process** (registers handlers for its group). The idiomatic worker is the bundled ace command, which boots the app and reads the same `config/durable.ts` â€” no adapter wiring to repeat:

```bash title="a separate worker process"
node ace durable:work --group=payments
```

<Callout type="info">
Both the engine and the `durable:work` worker resolve the adapter from the `connection` you named in `config/queue.ts`, so the driver + host live in exactly one place. For a non-Adonis setup you can still construct the primitive `QueueTransport` (exported from `@adonis-agora/durable`) by hand and hand it a raw `@adonisjs/queue` adapter via its `adapter` option.
</Callout>

## Options â€” `transports.queue(config)`

| Option | Default | Description |
| --- | --- | --- |
| `connection` | `config/queue.ts` `default` | Name of the `@adonisjs/queue` connection (a key of that file's `adapters`) whose adapter this transport uses. |
| `adapter` | â€” | Escape hatch: a raw `@adonisjs/queue` adapter factory, used instead of resolving from `config/queue.ts`. Takes precedence over `connection`. |
| `group` | â€” | The worker group this instance serves â€” required to register `handle()`s. |
| `prefix` | `'durable'` | Namespaces the queues (`<prefix>:tasks:<group>`, `<prefix>:results`, â€¦). |
| `pollIntervalMs` | `200` | Poll interval in ms between queue reads. |
| `instanceId` | random uuid | Stable process id stamped on heartbeats and control messages. |

`transports.queue(config)` returns the lazy driver thunk used in `config/durable.ts`; the underlying class is `QueueTransport`, also exported from `@adonis-agora/durable` for constructing a worker-side instance directly.

## Queue naming

| Queue | Direction | Purpose |
| --- | --- | --- |
| `<prefix>:tasks:<group>` | engine â†’ workers | step dispatch, per group |
| `<prefix>:results` | workers â†’ engine | step results |
| `<prefix>:heartbeats` | workers â†’ engine | liveness for `timeoutMs` steps |
| `<prefix>:control` | bidirectional | lifecycle events + cancellation (best-effort) |

Because the payload is the documented `RemoteTask` / `StepResult` JSON, multiple worker processes can share the same queues â€” scale out by running more `durable:work` processes that register the same group's handlers.
